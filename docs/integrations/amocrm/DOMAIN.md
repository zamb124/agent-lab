## Иерархия и связи

### 1. Компания → Контакт → Сделка

**Компания** (Company) - юридическое лицо или организация.

- Может содержать **множество контактов** (1:N)
- Может быть связана с **множеством сделок** (1:N)
- Отображается в карточках контактов и сделок

**Контакт** (Contact) - физическое лицо.

- Может принадлежать **одной компании** (N:1)
- Может быть связан с **множеством сделок** (N:M)
- Основная контактная информация: телефон, email, мессенджеры

**Сделка** (Lead) - процесс продажи.

- Может быть связана с **несколькими контактами** (N:M)
- Связана с **одной компанией** (N:1) через главный контакт
- Находится в **воронке** на **этапе**
- Имеет **ответственного** пользователя

### 2. Контакт → Покупатель

**Покупатель** (Customer) - контакт, который уже совершил покупку.

```
┌─────────────┐
│   Контакт   │
└──────┬──────┘
       │ 1:1
       ▼
┌─────────────┐
│ Покупатель  │───┐
└─────────────┘   │
                  │ 1:N
                  ▼
           ┌──────────────┐
           │  Транзакции  │
           │  (Purchases) │
           └──────────────┘
```

- Покупатель создается автоматически или вручную из контакта
- Имеет свой жизненный цикл (сегменты и статусы)
- Может совершать множественные покупки (транзакции)

### 3. Задачи

**Задача** (Task) - действие, которое нужно выполнить.

```
┌──────────────┐
│    Задача    │
└──────┬───────┘
       │
       ├──► Привязана к сущности (Lead/Contact/Company/Customer)
       ├──► Имеет тип (звонок, встреча, письмо)
       ├──► Имеет ответственного (User)
       └──► Имеет срок выполнения (deadline)
```

Задача может быть:
- Привязана к сделке, контакту, компании или покупателю
- Назначена на пользователя
- Имеет тип: звонок, встреча, письмо, задача

## Детальное описание сущностей

### Сделка (Lead)

**Основные свойства:**
- `id` - уникальный идентификатор
- `name` - название сделки
- `price` - бюджет сделки
- `responsible_user_id` - ответственный менеджер
- `status_id` - этап в воронке
- `pipeline_id` - воронка продаж
- `created_at`, `updated_at`, `closed_at` - временные метки
- `closest_task_at` - ближайшая задача
- `loss_reason_id` - причина закрытия (при отказе)

**Связи:**
- `_embedded.contacts[]` - связанные контакты
- `_embedded.companies[]` - связанная компания
- `_embedded.tags[]` - теги
- `custom_fields_values[]` - значения полей

**Жизненный цикл:**
```
Новая → Переговоры → Принятие решения → Согласование → Успешно закрыта
  │                                                          ▲
  └──────────────► Закрыта и не реализована ────────────────┘
                          (с указанием причины)
```

### Контакт (Contact)

**Основные свойства:**
- `id` - уникальный идентификатор
- `name` - ФИО контакта
- `first_name`, `last_name` - имя и фамилия
- `responsible_user_id` - ответственный менеджер
- `created_at`, `updated_at` - временные метки
- `closest_task_at` - ближайшая задача

**Связи:**
- `_embedded.companies[]` - связанная компания
- `_embedded.leads[]` - связанные сделки
- `_embedded.customers[]` - профиль покупателя
- `custom_fields_values[]` - контактные данные (телефон, email, мессенджеры)

### Компания (Company)

**Основные свойства:**
- `id` - уникальный идентификатор
- `name` - название компании
- `responsible_user_id` - ответственный менеджер
- `created_at`, `updated_at` - временные метки

**Связи:**
- `_embedded.contacts[]` - сотрудники компании
- `_embedded.leads[]` - сделки с компанией
- `custom_fields_values[]` - реквизиты, адрес, телефон

### Покупатель (Customer)

**Основные свойства:**
- `id` - уникальный идентификатор
- `name` - имя покупателя
- `next_price` - ожидаемая сумма следующей покупки
- `next_date` - ожидаемая дата следующей покупки
- `responsible_user_id` - ответственный менеджер
- `periodicity` - периодичность покупок (дни)
- `status_id` - статус покупателя

**Статусы покупателя:**
- Новый покупатель
- Постоянный покупатель
- Спящий покупатель
- Потерянный покупатель

**Транзакции:**
- Покупатель может иметь множество транзакций (покупок)
- Каждая транзакция связана с конкретной сделкой или создается отдельно

### Задача (Task)

**Основные свойства:**
- `id` - уникальный идентификатор
- `text` - описание задачи
- `complete_till` - срок выполнения (timestamp)
- `entity_id` - ID сущности (сделка/контакт/компания)
- `entity_type` - тип сущности (leads/contacts/companies/customers)
- `responsible_user_id` - ответственный
- `is_completed` - выполнена ли
- `task_type_id` - тип задачи
- `result` - результат выполнения

**Типы задач:**
- Звонок (call)
- Встреча (meeting)
- Письмо (email)
- Общая задача (task)

## Воронки и этапы

### Структура воронок

```
┌───────────────────────────────────────────────┐
│           Аккаунт AmoCRM                      │
└───────────────┬───────────────────────────────┘
                │
                │ содержит
                ▼
┌───────────────────────────────────────────────┐
│        Воронки (Pipelines)                    │
│  - Продажи (по умолчанию)                     │
│  - Входящие заявки                            │
│  - VIP-клиенты                                │
│  - Дополнительные воронки                     │
└───────────────┬───────────────────────────────┘
                │ 1:N
                ▼
┌───────────────────────────────────────────────┐
│          Этапы (Statuses)                     │
│  1. Неразобранное (Unsorted)                  │
│  2. Первичный контакт                         │
│  3. Переговоры                                │
│  4. Принятие решения                          │
│  5. Успешно реализовано (Successful)          │
│  6. Закрыто и не реализовано (Declined)       │
└───────────────────────────────────────────────┘
```

### Воронка продаж (Pipeline)

**Основные свойства:**
- `id` - уникальный идентификатор
- `name` - название воронки
- `sort` - порядок сортировки
- `is_main` - является ли основной
- `is_unsorted_on` - включено ли "Неразобранное"
- `_embedded.statuses[]` - этапы воронки

**Цифровая воронка (Digital Pipeline):**
- Автоматизированная воронка для работы с лидами из digital-каналов
- Может автоматически менять этапы при наступлении событий
- Настраивается администратором

### Неразобранное (Unsorted)

Специальная область для входящих заявок, которые еще не обработаны:

```
┌──────────────────────┐
│   Неразобранное      │
│     (Unsorted)       │
└──────┬───────────────┘
       │
       ├──► sip (звонки)
       ├──► mail (письма)
       ├──► forms (формы с сайта)
       ├──► chats (сообщения из чатов)
       └──► other (прочие)
```

После принятия заявки она автоматически превращается в сделку или контакт.

## Чаты и коммуникации

### Структура чатов

```
┌────────────────────────────────────────────┐
│              Чаты (Chats)                  │
└────────┬───────────────────────────────────┘
         │
         ├──► Внутренние чаты (между пользователями аккаунта)
         │    - Привязаны к сделке или покупателю
         │    - Отображаются в карточке
         │
         └──► Внешние чаты (с клиентами)
              - Telegram, WhatsApp, Facebook и др.
              - Привязаны к сделке, покупателю или контакту
              └──► Беседы (Talks/Conversations)
                   ├── ID беседы
                   ├── Статус (открыта/закрыта)
                   ├── Сообщения (Messages)
                   │   ├── Текст
                   │   ├── Автор
                   │   ├── Время
                   │   └── Тип (входящее/исходящее)
                   └── Метаданные
```

### Беседа (Talk/Conversation)

**Основные свойства:**
- `id` - уникальный идентификатор беседы
- `entity_id` - ID сущности (сделка/контакт/покупатель)
- `entity_type` - тип сущности
- `created_at` - время создания
- `updated_at` - время последнего сообщения
- `is_closed` - закрыта ли беседа
- `source` - источник (telegram, whatsapp, etc.)

**Сообщения в беседе:**
```
Беседа (Talk)
    │
    ├──► Сообщение 1 (входящее от клиента)
    ├──► Сообщение 2 (исходящее от менеджера)
    ├──► Сообщение 3 (входящее от клиента)
    └──► ...
```

### Внешние чаты и интеграции

Поставщиком внешнего чата может быть:
- **Встроенные интеграции:** Telegram, Facebook, Instagram, VK
- **Сторонние интеграции:** WhatsApp (через провайдеров)
- **Кастомные интеграции:** через Chat API (amojo.amocrm.ru)

## Права доступа

### Структура прав

```
┌────────────────────────────────────┐
│         Пользователи               │
│           (Users)                  │
└───────────┬────────────────────────┘
            │
            │ входят в
            ▼
┌────────────────────────────────────┐
│          Группы (Groups)           │
│  - Администраторы                  │
│  - Менеджеры                       │
│  - Наблюдатели                     │
└───────────┬────────────────────────┘
            │
            │ имеют
            ▼
┌────────────────────────────────────┐
│      Права (Permissions)           │
│  - Просмотр                        │
│  - Редактирование                  │
│  - Удаление                        │
│  - Экспорт                         │
└────────────────────────────────────┘
```

### Роли пользователей

**Администратор:**
- Полный доступ ко всем сущностям
- Настройка интеграций и воронок
- Управление пользователями и правами
- Доступ к настройкам аккаунта

**Менеджер:**
- Работа со своими сделками, контактами, задачами
- Возможность видеть сделки других (зависит от настроек)
- Создание и редактирование сущностей

**Наблюдатель (read-only):**
- Только просмотр сущностей
- Нет прав на редактирование и создание

### Видимость сущностей

Каждая сущность имеет:
- `responsible_user_id` - ответственный пользователь
- Настройки видимости группы
- Права на просмотр/редактирование по ролям

```
Сделка
  ├──► Ответственный: Менеджер А (полный доступ)
  ├──► Группа: Отдел продаж (просмотр)
  └──► Администраторы (полный доступ)
```

## Интеграции и источники

### Источники (Sources)

**Источник** - интеграция или сервис, который создает сделки в аккаунте.

```
┌────────────────────────────────────┐
│         Источники                  │
└───────────┬────────────────────────┘
            │
            ├──► Формы (Forms)
            │    - Виджет на сайте
            │    - Отдельные формы
            │
            ├──► Мессенджеры (Messengers)
            │    - Telegram
            │    - WhatsApp
            │    - Instagram
            │
            ├──► Email
            │    - Почтовые ящики
            │
            ├──► Телефония
            │    - Входящие звонки
            │
            ├──► Маркетплейсы
            │    - Avito
            │    - Яндекс.Маркет
            │
            └──► API интеграции
                 - Сторонние системы
```

**Иерархия источников:**
- Источники могут быть множественными (несколько номеров WhatsApp)
- Источники отображаются в разделе Digital Pipeline
- Используются для определения воронки при создании сделки

### Интеграции (Integrations)

**Типы интеграций:**

1. **Приватная интеграция**
   - Для одного аккаунта
   - Может содержать виджет (JS-код)
   - Использует OAuth 2.0

2. **Публичная интеграция**
   - Доступна всем пользователям AmoCRM
   - Размещена в каталоге интеграций
   - Проходит модерацию
   - Может содержать виджет

3. **Внешняя интеграция**
   - Подключается из сторонней системы
   - Не содержит виджет
   - Только API доступ

4. **Отраслевое решение**
   - Специализированная интеграция для отрасли
   - Расширенный функционал

**Права интеграций:**
- Администратор может видеть все подключенные интеграции
- Пользователь может подключить интеграцию через OAuth
- Администратор или пользователь может отозвать доступ

```
Интеграция
  ├──► OAuth Credentials (client_id, client_secret)
  ├──► Виджет (widget.js) - опционально
  ├──► Scopes (права доступа)
  │    - crm.read
  │    - crm.write
  │    - leads.read
  │    - etc.
  └──► Access Tokens (по пользователям)
```

## Поля и каталоги

### Кастомные поля (Custom Fields)

Каждая сущность может иметь кастомные поля:

```
┌────────────────────────────────────┐
│      Типы полей                    │
└───────────┬────────────────────────┘
            │
            ├──► Текстовое (text)
            ├──► Числовое (numeric)
            ├──► Дата (date, datetime)
            ├──► Переключатель (checkbox)
            ├──► Список (select, multiselect)
            ├──► URL (url)
            ├──► Адрес (streetaddress)
            ├──► Телефон (phone)
            ├──► Email (email)
            ├──► Связь с каталогом (catalog)
            └──► Файл (file)
```

### Каталоги (Catalogs)

**Каталог** - справочник данных (например, товары, услуги).

```
┌────────────────────────────────────┐
│      Каталог (Catalog)             │
│      - Товары                      │
│      - Услуги                      │
└───────────┬────────────────────────┘
            │ 1:N
            ▼
┌────────────────────────────────────┐
│  Элементы каталога (Catalog Items)│
│  - Товар А (цена, артикул)         │
│  - Товар Б (цена, артикул)         │
└────────────────────────────────────┘
```

**Связь со сделками:**
- Сделка может содержать элементы из каталога
- Через поле типа `catalog` или прямую связь
- Автоматический подсчет суммы сделки

## События и аудит

### События (Events)

AmoCRM фиксирует все изменения в системе:

```
Сущность (Lead/Contact/Company)
    │
    └──► События (Events)
         ├──► Создание
         ├──► Изменение статуса
         ├──► Изменение полей
         ├──► Добавление примечания
         ├──► Выполнение задачи
         ├──► Отправка сообщения
         └──► Удаление
```

**Типы событий:**
- `entity_added` - создание сущности
- `entity_updated` - обновление сущности
- `entity_deleted` - удаление сущности
- `entity_linked` - связывание сущностей
- `status_changed` - смена этапа
- `task_completed` - выполнение задачи
- `note_added` - добавление примечания
- `call_incoming` - входящий звонок
- `message_sent` - отправка сообщения

### Примечания (Notes)

**Примечание** - запись о событии или действии.

**Типы примечаний:**
- `common` - обычное текстовое примечание
- `call_in` - входящий звонок
- `call_out` - исходящий звонок
- `extended_service_message` - системное сообщение от интеграции
- `sms_in` - входящее SMS
- `sms_out` - исходящее SMS
- `message_cashier` - сообщение от кассы

```
Сделка/Контакт
    │
    └──► Примечания (Notes)
         ├──► [12:30] Звонок от клиента (3 мин)
         ├──► [13:00] Отправлено КП на email
         ├──► [14:15] Клиент согласовал условия
         └──► [15:00] Перенесена на этап "Согласование"
```

## Webhooks

AmoCRM может отправлять уведомления о событиях на внешний URL:

```
AmoCRM Event (изменение сделки)
         │
         ▼
    Webhook URL (ваш сервер)
         │
         └──► Обработка события в вашей системе
```

**Доступные события для webhook:**
- `add` - создание сущности
- `update` - обновление сущности
- `delete` - удаление сущности
- `status` - смена статуса сделки
- `note` - добавление примечания
- `responsible` - смена ответственного

**Настройка:**
- Администратор создает webhook через API
- Указывает URL и типы событий
- AmoCRM отправляет POST запрос при событии

## Резюме

### Основная иерархия

```
Аккаунт
  │
  ├──► Пользователи + Группы
  │

  │     └──► Этапы

  │                 ├──► Контакты
  │                 │     └──► Компания
  │                 ├──► Задачи
  │                 ├──► Примечания
  │                 ├──► Чаты/Беседы

  │     └──► Интеграции
  │
  └──► События
```

### Ключевые принципы

1. **Все строится вокруг сделок** - основная единица работы в CRM
2. **Контакты и компании** - связаны со сделками (N:M)
3. **Покупатели** - отдельная сущность для постпродажного обслуживания
4. **Задачи** - могут быть привязаны к любой сущности
5. **Чаты** - интегрируются с внешними мессенджерами
6. **Воронки** - определяют жизненный цикл сделки
7. **Права доступа** - через группы и роли пользователей
8. **Интеграции** - расширяют функционал через API и виджеты
