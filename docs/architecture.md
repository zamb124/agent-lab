# Архитектура Agent Lab

Agent Lab — это платформа для создания и управления ИИ-агентами, разработанная с использованием подхода **Database-First** и паттерна **BFF (Backend for Frontend)**.

## Принципы архитектуры

1. **Database-First**: Вся конфигурация агентов, Flow и инструментов хранится в БД. Код определяет только поведение, но не структуру.
2. **BFF (Backend for Frontend)**: Сервис `Frontend Service` выступает как единственная точка входа для браузерного клиента, обеспечивая безопасность и удобство взаимодействия.
3. **Shared DB**: Сервисы `Agents` и `Frontend` используют общую базу данных для эффективной работы с данными без лишних HTTP-запросов.
4. **Единообразие**: Агенты, созданные через код и через пользовательский интерфейс, работают идентично.
5. **Асинхронность**: Вся архитектура полностью асинхронна. Используются `async/await` для всех операций.

## Компоненты системы

```mermaid
graph LR
    Browser[Браузерный клиент] -->|HTTPS/WSS| Frontend[Frontend Service (8002)]
    Frontend -->|Репозитории| DB[(База данных)]
    Frontend -->|Внутренний HTTP| Agents[Agents Service (8001)]
    Agents -->|Репозитории| DB
    Agents -->|Воркер| Tasks[Очередь задач]
    
    ExternalApp[Внешний API Клиент] -->|HTTPS| Agents
```

### Frontend Service (8002)
- **Роль:** BFF (Backend for Frontend) и UI.
- **Ответственность:**
  - Отдача статики и HTML (Jinja2, HTMX).
  - API для UI (`/frontend/api`).
  - Прямая работа с БД для CRUD операций (чтение/запись конфигураций) через `AgentsContainer`.
  - Проксирование/Делегирование выполнения тяжелых задач в `Agents Service` (через БД Tasks или HTTP).
- **Доступ к данным:** Использует общие репозитории (`AgentRepository`, `FlowRepository`) для прямого доступа к БД.

### Agents Service (8001)
- **Роль:** Исполнитель бизнес-логики и Внешнее API.
- **Ответственность:**
  - Выполнение агентов и инструментов.
  - Обработка фоновых задач (`TaskProcessor`).
  - Внешнее API (`/agents/api/v1`) для интеграций.
- **Безопасность:** Закрыт от прямых запросов из браузера (CORS отключен/пустой). Доступен только для Frontend Service (server-to-server) и внешних API клиентов (не браузеров).

## Взаимодействие

### 1. CRUD Операции (Чтение/Запись конфигурации)
**Поток:** UI -> Frontend Service -> Repository -> DB.

Frontend Service использует `AgentsContainer` для прямого доступа к репозиториям. Это обеспечивает высокую производительность и отсутствие лишних сетевых задержек.

### 2. Выполнение Агента (Чат)
**Поток:** UI -> Frontend Service -> DB (Task) -> Agents Service (Worker).

Чат работает асинхронно. Frontend Service создает задачу в БД. Воркер Agents Service подхватывает задачу, выполняет агента и сохраняет результат/отправляет уведомление через WebSocket.

### 3. Синхронное Выполнение (если требуется)
**Поток:** UI -> Frontend Service -> HTTP -> Agents Service.

Если требуется синхронное выполнение тяжелой логики (например, тест инструмента), Frontend Service может сделать HTTP запрос к Agents Service. Но предпочтительным является асинхронный подход через задачи.

## Структура проекта

```
agent-lab/
├── core/                          # Общая инфраструктура
│   ├── db/                        # БД и Репозитории
│   ├── container/                 # DI Контейнер
│   └── ...
├── apps/
│   ├── agents/                    # Сервис агентов (8001)
│   │   ├── api/                   # Внешнее API
│   │   ├── agents/                # Логика агентов
│   │   └── container.py           # AgentsContainer
│   └── frontend/                  # Сервис фронтенда (8002)
│       ├── api/                   # BFF API
│       ├── modules/               # UI Модули
│       └── container.py           # FrontendContainer
```

## Развертывание

В продакшене используется Nginx для маршрутизации:
- `/frontend/*` -> 8002
- `/agents/*` -> 8001 (для внешних API)

Локально для разработки доступны оба порта (8001, 8002), но UI должен взаимодействовать только с 8002.
