
## **Архитектура Динамической, Управляемой Агентом Системы Планирования Целей**

**Версия 1.0**

### **1. Обзор и Философия**

Данная архитектура описывает создание интеллектуальной системы, способной динамически планировать, выполнять и адаптировать многошаговые жизненные цели пользователя (например, путешествия, накопления, ремонт).

**Ключевая философия:** система разделена на два фундаментальных уровня:
1.  **Уровень Интеллекта (Агент, "Архитектор и Прораб"):** Отвечает за **ЧТО** делать. Он создает и изменяет детальный "чертеж" (План) цели. Этот уровень гибок, умен и адаптивен.
2.  **Уровень Исполнения (Temporal, "Надежный Строитель"):** Отвечает за **КАК** это делать. Он тупо, но чрезвычайно надежно следует "чертежу" (Плану), выполняя каждый шаг, управляя временем и сбоями. Этот уровень надежен, отказоустойчив и наблюдаем.

Такое разделение позволяет сочетать непредсказуемую гибкость больших языковых моделей с инженерной надежностью систем оркестрации.

### **2. Ключевые Компоненты**

| Компонент | Технология | Ответственность |
| :--- | :--- | :--- |
| **Приложение (Backend)** | Python/Go/Node.js | "Приемная". Принимает запросы, определяет, новая это цель или существующая, и вызывает соответствующего агента ("Конструктора" или "Маршрутизатора"). |
| **Система Оркестрации** | **Temporal** | "Исполнитель". Запускает и управляет одним универсальным, но надежным Воркфлоу (`DynamicPlanExecutor_Workflow`), который читает и выполняет План. |
| **Система ИИ** | **LangChain/LangGraph** | "Мозг". Содержит агентов, которые генерируют и обновляют План. Выполняется внутри `Activities`, вызываемых из Temporal. |

### **3. Центральная Сущность: Динамический План (JSON)**

План — это сердце всей системы. Это структурированный JSON-документ, который является единственным источником правды о состоянии цели. Он хранится внутри состояния Воркфлоу в Temporal.

**Детальная структура Плана:**
```json
{
  "goal_name": "Путешествие в Японию, весна 2026",
  "current_step_id": "step_2_search_tickets", // ID шага, который выполняется сейчас или будет выполняться следующим
  "status": "IN_PROGRESS", // Статус всей цели: PENDING, IN_PROGRESS, COMPLETED, FAILED, CANCELLED
  "context": { // Общий контекст, который могут использовать все шаги
    "destination": "Япония",
    "budget": 350000,
    "dates": { "start": "2026-04-10", "end": "2026-04-24" }
  },
  "steps": [
    {
      "id": "step_1_clarify_details",
      "name": "Уточнение деталей поездки у пользователя",
      "type": "USER_INTERACTION", // Тип шага
      "status": "COMPLETED", // Статус шага: PENDING, IN_PROGRESS, COMPLETED, FAILED, SKIPPED
      "result": { // Результат выполнения шага
        "budget_confirmed": true,
        "dates_confirmed": true
      }
    },
    {
      "id": "step_2_search_tickets",
      "name": "Поиск и выбор авиабилетов",
      "type": "TOOL_CALL",
      "tool_name": "search_flights_tool", // Какую Activity вызывать
      "params": { // Параметры для Activity
        "destination_code": "NRT",
        "dates": { "$ref": "context.dates" } // Ссылка на данные из общего контекста
      },
      "status": "IN_PROGRESS",
      "depends_on": ["step_1_clarify_details"] // Шаг не начнется, пока не завершится зависимость
    },
    {
      "id": "step_3_book_hotel",
      "name": "Бронирование отеля",
      "type": "TOOL_CALL",
      "tool_name": "book_hotel_tool",
      "status": "PENDING",
      "depends_on": ["step_2_search_tickets"]
    },
    {
      "id": "step_4_remind_checkin",
      "name": "Напомнить о регистрации на рейс",
      "type": "SCHEDULED_TASK",
      "status": "PENDING",
      "depends_on": ["step_2_search_tickets"],
      "run_at": { // Динамическое время выполнения
        "source_step": "step_2_search_tickets",
        "source_field": "result.flight_details.departure_time",
        "offset": "-24h" // За 24 часа до времени вылета
      }
    }
  ]
}
```

### **4. Комплексная BPMN-схема Процесса**

```mermaid
bpmn
    participant Пользователь

    participant "Приложение (Backend)" as BE
        lane "API / Проверка ID" as API
        lane "Агент-Конструктор Плана" as CONSTRUCTOR
        lane "Агент-Маршрутизатор" as ROUTER

    participant "Система Оркестрации (Temporal)" as TE
        lane "Воркфлоу-Исполнитель" as WF
        lane "Обработчик Сигналов" as SH

    participant "Система ИИ (LangGraph Worker)" as AI
        lane "Агент-Обновитель Плана" as UPDATER

    %% --- Path 1: New Goal ---
    Пользователь->>API: [Сообщение: "Хочу в Японию!"]
    activate API
        API->>API: Задача: Проверить наличие Workflow ID
        API-->>CONSTRUCTOR: Шлюз: ID не найден
    deactivate API
    activate CONSTRUCTOR
        CONSTRUCTOR->>CONSTRUCTOR: Задача: Сгенерировать первоначальный План (JSON)
        CONSTRUCTOR-->>WF: [Сообщение: Команда startWorkflow(initial_plan)]
    deactivate CONSTRUCTOR

    %% --- Workflow Execution Loop ---
    activate WF
        WF->>WF: Задача: Найти следующий PENDING шаг в Плане
        WF->>WF: Шлюз: Какой тип шага?
        WF-->>Пользователь: Тип: USER_INTERACTION
        WF->>WF: [Сообщение: Вопрос пользователю]
        WF->>WF: Событие: Ожидать сигнал "userInput"
        note right of WF: Воркфлоу "заснул"
    deactivate WF

    %% --- Path 2: User Response / Interrupt ---
    Пользователь->>API: [Сообщение: "Бюджет 350к" / "Передумал, хочу в Корею"]
    activate API
        API->>API: Задача: Проверить наличие Workflow ID
        API-->>ROUTER: Шлюз: ID найден
    deactivate API
    activate ROUTER
        ROUTER->>ROUTER: Задача: Сгенерировать Сигнал (напр., "userInput" или "updatePlanByUser")
        ROUTER-->>SH: [Сообщение: Команда signalWorkflow(signal)]
    deactivate ROUTER

    %% --- Signal Handling and Plan Update ---
    activate SH
        SH->>SH: Событие: Получен сигнал
        SH-->>UPDATER: [Сообщение: Запрос "Обнови План" (текущий_план + сигнал)]
    deactivate SH
    activate UPDATER
        UPDATER->>UPDATER: Задача: Проанализировать и сгенерировать НОВЫЙ План (JSON)
        UPDATER-->>WF: [Сообщение: Ответ "Новый План"]
    deactivate UPDATER
    activate WF
        WF->>WF: Задача: Заменить старый План на новый
        note right of WF: Воркфлоу "проснулся" и готов<br/>исполнять обновленный план
        WF->>WF: Задача: Найти следующий PENDING шаг... (цикл продолжается)
    deactivate WF
```

### **5. Детальный разбор сценариев**

#### **Сценарий A: Создание новой цели (Happy Path)**

1.  **Пользователь -> API:** Отправляет первое сообщение: `"Хочу поехать в Японию весной 2026, помоги спланировать"`.
2.  **Backend (API -> Конструктор):**
    *   Проверяет БД и не находит активного `workflow_id` для этой цели.
    *   Вызывает **Агента-Конструктора Плана**.
    *   Агент рассуждает и генерирует детальный первоначальный JSON-план (как в примере выше), где первый шаг — уточнение деталей (`USER_INTERACTION`).
    *   Бэкенд запускает в Temporal `DynamicPlanExecutor_Workflow`, передавая этот План как стартовый параметр.
3.  **Temporal (Воркфлоу-Исполнитель):**
    *   Воркфлоу стартует. Он не вызывает агента.
    *   Он читает свой `plan.json`, находит первый шаг (`step_1_clarify_details`) со статусом `PENDING`.
    *   Он видит, что тип шага `USER_INTERACTION`. Он запускает `Activity`, которая отправляет пользователю вопрос ("Давайте уточним бюджет и даты?"), и сразу же вызывает `workflow.waitForSignal("userInput")`, "засыпая" до ответа.

#### **Сценарий B: Пользователь отвечает и План выполняется**

1.  **Пользователь -> API:** Отвечает: `"Да, бюджет 350 тысяч, даты с 10 по 24 апреля"`.
2.  **Backend (API -> Маршрутизатор):**
    *   Находит `workflow_id` в БД.
    *   Вызывает **Агента-Маршрутизатора**, который классифицирует сообщение как ответ на вопрос и формирует сигнал `userInput` с данными ответа.
    *   Отправляет сигнал в работающий Воркфлоу.
3.  **Temporal (Обработчик Сигналов -> Обновитель Плана -> Исполнитель):**
    *   `Обработчик Сигналов` "пробуждает" Воркфлоу.
    *   Он вызывает **Агента-Обновителя Плана** с задачей: "Вот текущий План, вот ответ пользователя. Обнови План".
    *   Агент обновляет JSON: меняет статус `step_1` на `COMPLETED`, записывает подтвержденные данные в `context` и, возможно, уточняет параметры для следующих шагов.
    *   Воркфлоу заменяет свой старый План на новый.
    *   Цикл исполнения продолжается. Воркфлоу ищет следующий `PENDING` шаг. Находит `step_2_search_tickets`.
    *   Он видит, что тип шага `TOOL_CALL`. Он вызывает `Activity` `search_flights_tool` с параметрами из Плана. После успешного выполнения он обновит статус `step_2` на `COMPLETED` и перейдет к `step_3`.

#### **Сценарий C: Пользователь прерывает процесс и меняет цель**

1.  **Пользователь -> API:** Пишет в любой момент: `"Стоп, я передумал. Япония слишком дорого. Давай лучше в Таиланд"`.
2.  **Backend (API -> Маршрутизатор):**
    *   Находит `workflow_id`.
    *   Маршрутизатор понимает, что это не ответ, а кардинальное изменение цели. Он формирует сигнал `updatePlanByUser` с текстом пользователя.
3.  **Temporal (Обработчик Сигналов -> Обновитель Плана -> Исполнитель):**
    *   `Обработчик Сигналов` перехватывает сигнал.
    *   Он вызывает **Агента-Обновителя Плана** с задачей: "Вот текущий План, вот новый ввод от пользователя. Требуется полное перепланирование".
    *   **Агент-Обновитель Плана** рассуждает: "Цель изменилась. Старый план неактуален. Нужно сгенерировать совершенно новый план для поездки в Таиланд".
    *   Он создает **полностью новый JSON-план** и возвращает его.
    *   Воркфлоу **выбрасывает свой старый План и заменяет его новым**.
    *   Цикл исполнения начинается заново с первого шага нового Плана.

#### **Сценарий D: Обработка отложенной задачи (Напоминание)**

1.  **Контекст:** Воркфлоу дошел до шага `step_4_remind_checkin`. Его статус `PENDING`.
2.  **Temporal (Воркфлоу-Исполнитель):**
    *   Он видит, что тип шага `SCHEDULED_TASK`.
    *   Он читает поле `run_at`. Он видит, что время зависит от результата шага `step_2`.
    *   Он извлекает время вылета из `plan.steps[1].result.flight_details.departure_time`, вычитает 24 часа и получает точное время для пробуждения.
    *   Он вызывает `workflow.sleep(duration)`. Воркфлоу "засыпает" на несколько дней/недель.
    *   Когда таймер срабатывает, Воркфлоу "просыпается", обновляет статус `step_4` на `COMPLETED` и ищет следующий шаг.

#### **Сценарий E: Обработка ошибки инструмента**

1.  **Контекст:** Воркфлоу пытается выполнить `step_3_book_hotel`, вызывая `Activity` `book_hotel_tool`.
2.  **Temporal:** `Activity` падает (например, API отеля недоступен). Temporal, согласно своей политике `retry`, автоматически пытается выполнить ее снова несколько раз.
3.  **Если все попытки провалились:**
    *   `Activity` возвращает ошибку в Воркфлоу.
    *   Воркфлоу обновляет статус `step_3` на `FAILED` и записывает в `result` информацию об ошибке.
    *   **Важно:** Воркфлоу не падает. Он вызывает **Агента-Обновителя Плана** с задачей: "Шаг `step_3` провалился с такой-то ошибкой. Обнови План".
4.  **Агент-Обновитель Плана:**
    *   Рассуждает: "API бронирования не работает. Я могу: а) добавить новый шаг, чтобы попробовать другой API; б) добавить шаг `USER_INTERACTION`, чтобы спросить пользователя, что делать; в) отменить цель".
    *   Он выбирает вариант "б" и генерирует новый План, где добавлен шаг с вопросом: `"К сожалению, сервис бронирования временно недоступен. Попробовать позже или поискать отели на другом сайте?"`.
    *   Воркфлоу получает этот новый План и продолжает исполнение, задавая вопрос пользователю.

### **6. Преимущества Архитектуры**

*   **Максимальная Гибкость:** Агент может создавать и изменять планы любой сложности, не будучи ограниченным жестким кодом воркфлоу.
*   **Полная Наблюдаемость (Observability):** JSON-план в состоянии Воркфлоу — это идеальный лог и источник правды. В любой момент видно, что было, что есть и что будет.
*   **Исключительная Надежность:** Temporal гарантирует, что каждый шаг плана будет выполнен надежно, с повторными попытками, а весь процесс переживет любые сбои серверов.
*   **Четкое Разделение Ответственности:** Код (Temporal) отвечает за надежность. Данные (План) и логика их генерации (Агент) отвечают за интеллект.
