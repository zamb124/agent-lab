# План реализации по каталогу `mobile/`

Канон фаз и ворот: `[mobile/docs/WATERFALL.md](../../mobile/docs/WATERFALL.md)`. Ниже — **порядок исполнения** и чеклисты, с которыми можно заходить в реализацию по шагам.

---

## Принцип

- Веб не дублируется: оболочки грузят production URL.
- Репозиторий уже содержит PWA (`[core/frontend/pwa/](../../core/frontend/pwa/)`, `[core/app/pwa_routes.py](../../core/app/pwa_routes.py)`); сборки — в `[mobile/](../../mobile/)`.
- Ключи подписи, аккаунты Apple/Google, оплата — **вне git** (`[mobile/config/env.example](../../mobile/config/env.example)`, секреты в CI).

## Фаза 1 — Инициация (параллельно с подготовкой людей)


| Шаг | Действие                                                                          | Готово когда                 |
| --- | --------------------------------------------------------------------------------- | ---------------------------- |
| 1.1 | Зафиксировать список магазинов v1 (минимум Play + App Store)                      | Документ/таблица согласована |
| 1.2 | Матрица: WebRTC, WS, push, файлы, голос — «ок / ограничение» по Android TWA и iOS | Заполнена                    |

**Код репозитория:** не обязателен на этой фазе.

## Фаза 2 — Готовность веба (блокер для 3 и 4)


| Шаг | Действие                                                                                                               | Артефакты / пути                                                                                         |
| --- | ---------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| 2.1 | Production: HTTPS, манифест отдаётся 200                                                                               | `curl`/браузер на `PWA_MANIFEST_URL`                                                                     |
| 2.2 | Lighthouse (или CI) на prod: installability, SW, manifest                                                              | Отчёт; при необходимости правки в `[core/frontend/pwa/](../../core/frontend/pwa/)`                       |
| 2.3 | Решение по tenant: один вход для оболочки (`start_url` / `scope`)                                                      | Запись в тикете или короткий ADR; см. `[tenant-url](../../core/frontend/static/lib/utils/tenant-url.js)` |
| 2.4 | Проверка VAPID: `GET {service}/api/push/vapid-public-key` на том же origin                                             | Ручной тест в браузере                                                                                   |
| 2.5 | Локально: `cd mobile && npm install`, скопировать `config/env.example` → `.env`, `./scripts/check-pwa-manifest-url.sh` | Скрипт завершается OK                                                                                    |


**Ворота Фазы 2:** нет блокирующих замечаний аудита; зафиксирован `PWA_MANIFEST_URL`.

## Следующий шаг для агента/разработчика

1. Закрыть **Фазу 1** на уровне команды (таблица магазинов + матрица).
2. Выполнить **Фазу 2** в проде и зафиксировать `mobile/.env`.
3. Перейти к **Фазе 3** или **4** по инструкциям в `mobile/android/README.md` и `mobile/ios/README.md`.

При явном запросе «реализуй Фазу N» — выполнять только задачи этой фазы и ворота WATERFALL.
